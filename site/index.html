When a model's experts fit the budget, Sparsify loads them fully and runs the original computation — measured at parity with unmodified mlx-lm (154 vs 151.5 tok/s). When they don't fit, decode is SSD-bound: expert misses stream from disk at the device's random-read speed (~0.5 GB/s on the test USB SSD). Correctness holds in both modes — paged output is token-identical to full-RAM inference, verified with evictions active.

how it works

Virtual memory, applied to intelligence

No forks, no model conversion, no architecture-specific code. Sparsify detects expert structure from the model itself and swaps in paged projections at the leaf level.

1 · load

Lazy open

The model loads with zero weight bytes read from disk.

2 · detect

Find the experts

Any projection with a leading expert dimension is detected structurally — Mixtral, Qwen-MoE, OLMoE, DeepSeek, future models.

3 · swap

Surgery

Expert projections become paged modules. Router, attention, everything else: untouched upstream code.

4 · page

Read on route

When the router picks experts, exactly those weight slices are read — one contiguous pread each — into a byte-budgeted LRU cache.

5 · bound

Evict & measure

Inactive experts leave RAM. Hits, misses, evictions, SSD bytes and latency stream in live telemetry.

docs · quickstart

Five commands, end to end

Install & run

# one-command install (creates ~/.sparsify, adds `sparsify` to PATH)
$ curl -fsSL https://github.com/daylinkltd/sparsify/releases/latest/download/install.sh | sh

# browse the catalog, pull a model (idempotent, resumable)
$ sparsify models
$ sparsify pull qwen:30b-a3b

# chat — RAM budget sizes itself from free memory
$ sparsify run qwen:30b-a3b

# or pin the expert cache explicitly
$ sparsify run qwen:30b-a3b --memory-limit 3

# OpenAI-compatible server
$ sparsify serve qwen:30b-a3b

# or run it as a login service, Ollama-style (localhost:7777)
$ sparsify start
$ curl localhost:7777/v1/chat/completions \
    -d '{"model":"qwen:30b-a3b","messages":[{"role":"user","content":"hi"}]}'

the honest part

What's proven, what's next

Proven: correctness (paged output identical to full-RAM inference, verified with active evictions) and bounded memory (RSS = backbone + your budget, flat, regardless of model size).

Not yet solved: decode speed when the expert working set is far larger than the cache. Qwen3-30B with a 4.5 GB budget runs at 2.6 tok/s on a USB SSD (96.9% cache hits; the misses are real routing churn — we replayed a 129k-access trace against LFU, CLOCK and segmented-LRU, and none beat LRU). Expert reads are parallelized to the device's measured ceiling; past that, speed scales with your SSD and your budget. Those numbers live in our telemetry, on purpose, in public.

The claim we stand behind: stored intelligence should be limited by your SSD, not your RAM — and you get the model's exact intelligence, not an approximation.

  • Done: parallel expert reads (3.7× decode); hybrid residency — models that fit the budget run at native mlx-lm speed
  • Async prefetch: read the next block's experts while the current one computes
  • Per-layer budget partitioning for working sets far beyond RAM (Mixtral-class)
  • Routing-trace telemetry to ground prefetch policy in measured locality
  • GGUF store backend; KV-cache persistence across chat turns

frontier

CUDA backend

The paging core (store, cache, surgery) is backend-agnostic by design. Bringing Linux/Windows online is the single highest-impact contribution.

Start here: github.com/daylinkltd/sparsify — CONTRIBUTING.md has the ground rules (short version: measure everything, label everything, no simulated benchmarks).
